# 역(逆)팔란티어 — 제품 전략 노트

> **한 줄**: 목표 온톨로지(원하는 운영 모델)를 먼저 선언하고, 그것을 실현할 데이터를
> 시스템이 거꾸로(demand-pull) 끌어모으는 운영 OS. — 팔란티어와 정반대 방향.

이 문서는 FlowDesk의 중장기 방향을 규정한다. 기능 명세(SPEC.md)·결정 로그(DECISIONS.md)와 함께 읽는다.

---

## 1. 문제: 팔란티어 모델은 SMB에 안 맞는다

팔란티어(Foundry/AIP)의 가치는 **이미 존재하는, 여러 시스템에 흩어진 대량의 데이터**를
온톨로지(의미 계층)로 통합·정리하는 데서 나온다. 그래서 전제가 **"데이터가 이미 있다"**.

우리 타깃(FDE를 쓸 여력이 없는 중소기업)은 그 전제가 깨진다:
- 데이터가 흩어져 있고(엑셀·카톡·수기·SaaS 몇 개), 양이 적으며,
- 상당수가 아예 시스템 밖(사람 머릿속)에 있다.

→ 따라서 **"통합부터 / 온톨로지로 정리부터"는 SMB에 과하거나 불가능**하다.

---

## 2. 해법: 방향을 뒤집는다

| | 출발점 | 방향 | 데이터 |
|---|---|---|---|
| **팔란티어** | 있는 데이터 | 데이터 → **온톨로지로 정리** → 운영 | 이미 존재 (supply-push) |
| **FlowDesk (역방향)** | 원하는 운영 모델 | **온톨로지 선언** → 데이터를 **확보** → 실현 | 확보해서 생성 (demand-pull) |

핵심 통찰 두 가지:

1. **데이터가 없어도 온톨로지는 정의할 수 있다.**
   온톨로지는 "내가 무엇을 가졌나"가 아니라 **"내 업무엔 어떤 객체·정보·결정이 의미 있나"**
   이므로, 빈손인 SMB도 *원하는 모습*으로 먼저 그릴 수 있다.
   (우리의 **자연어 → 프로세스/온톨로지** 기능이 정확히 이 선언을 FDE 없이 수행.)

2. **온톨로지 = 수요(demand)의 명세서.** 객체·필드·관계·액션을 선언하면, 그것은 곧
   "이 모델을 현실로 만들려면 어떤 데이터가 필요한가"의 정의가 된다. 시스템의 임무는
   그 **빈 칸을 채울 데이터를 가서 확보**하는 것.

> **온톨로지 = 빈 칸이 정의된 양식. 데이터 확보 = 그 빈 칸을 어떤 수단으로든 메우는 일.**

---

## 3. 데이터 확보 엔진 (우리만의 차별 역량)

온톨로지가 수요를 정의하면, 그걸 채우는 **공급 채널**이 여럿 붙는다. 팔란티어가 "통합"이라면
우리는 **"생성·확보"** 가 핵심 역량이다.

| # | 채널 | 설명 | 재사용하는 기존 자산 |
|---|---|---|---|
| 1 | **프로세스 실행** | 업무를 모델 위에 올려 돌리면 진행 위해 필요한 데이터가 빨려 들어옴 (주 채널) | 프로세스 엔진 · 폼 |
| 2 | **사람에게 요청** | 빠진 정보는 보유자에게 "찾아오기 과제"로 라우팅 | 협조업무 · 업무지시 · 결재 |
| 3 | **AI 탐색·제안** | 과거 인스턴스·업무일지·문서에서 후보값을 끌어와 채움/제안 | AI(Claude) |
| 4 | **경량 임포트** | 엑셀/회계 등 *있는 것*은 끌어와 채움 (여러 채널 중 하나일 뿐) | (신규 커넥터) |
| 5 | **백필** | 과거/진행 중 업무를 소급 입력해 과거 데이터도 구조화 | 프로세스 인스턴스 |

**미충족 정보 탐지 → 수집 유도**가 엔진의 핵심 루프:
노드가 요구하는 입력이 비면 "이 단계 진행에 X가 필요하나 아직 없음" → 진행 보류 + 위 채널로 확보.

## 4. 개념 모델

```
[온톨로지 / 목표 모델]  ← 먼저 선언 (자연어로도 가능, 데이터 0이어도 됨)
   ├─ 객체 타입 (고객, 계약, 설비, 품목 …)
   ├─ 필드 / 관계
   └─ 액션 (객체에 대한 쓰기 작업; 검증·승인 포함)
          │  "이 모델을 실현하려면 이런 데이터가 필요" = 수요 정의
          ▼
[데이터 확보 엔진]  ← 빈 칸을 채운다 (demand-pull)
   ├─ 프로세스 실행(주) · 사람 요청 · AI 탐색 · 임포트 · 백필
   └─ 미충족 정보 탐지 → 수집 라우팅
          │
          ▼
[실현된 운영 데이터]  ← 온톨로지가 데이터로 채워져 살아있는 상태
   └─ 그 위에서 AI 질의 · 대시보드 · 의사결정 폐루프
```

우리 프로세스 노드(TASK/APPROVAL)는 **"온톨로지 객체에 대한 액션"** 으로 진화하고,
우리 결재 모듈은 그 **액션의 승인(휴먼인더루프) 계층**이 된다. 즉 기존 엔진을 버리지 않고,
팔란티어의 "온톨로지 액션 + 승인"을 SMB가 쓰기 쉽게 만든 형태로 재포지셔닝한다.

## 5. 빌드 함의 (우선순위)

1. **온톨로지 정의 계층 (목표 모델 spec)**
   - 테넌트가 객체 타입·필드·관계·액션을 **선언**(노코드 + 자연어 보조).
   - "기존 데이터 통합용"이 아니라 **"원하는 모델 선언용"**.
   - 데이터 모델: `ObjectType / ObjectField / ObjectLink / ObjectAction / ObjectInstance`.
   - `tenantId` 격리·감사로그·승인 모듈을 그대로 재사용.

2. **데이터 확보 엔진**
   - 미충족 칸 탐지 + 위 5개 채널로 채우기.
   - 프로세스 노드가 객체를 read/write(액션) + 빈 입력 시 수집 유도.
   - 재사용 엔티티(autocomplete): 같은 "고객 A"를 매번 재입력하지 않고 객체로 연결.

3. **그 위의 가치**
   - 채워진 운영 데이터에 대한 **AI 질의(테넌트 격리)** + **결재 액션 연결**.
   - 객체 중심 뷰("고객 360") · 대시보드 · 병목/사이클 분석.

> 단일 최대 레버는 **①(온톨로지 선언) + ②(확보 엔진)의 결합**이다. 온톨로지만 있고 확보가
> 없으면 빈 스키마, 확보만 있고 온톨로지가 없으면 방향 없는 데이터.

---

## 6. 리스크와 피드백 루프

- **이상론 리스크**: 온톨로지를 *원하는 대로* 먼저 그리면 현실과 어긋날 수 있다.
  → **돌려보며 안 맞으면 모델을 고치는 루프** 필요. (`definition.version` / 온톨로지 버전)
- **입력 마찰**: 진행을 위해 데이터를 넣어야 하므로 초기 부담.
  → AI 자동완성·과거값 재사용·협조 라우팅으로 마찰 최소화.
- **데이터 품질/중복**: 같은 실세계 엔티티가 중복 입력.
  → 재사용 객체(연결) + 중복 탐지.
